[서비스 이름] 소금빵 카페 베통 "오늘의 주문" 서비스

[대표 문구]
오늘만 굽는 소금빵, 오늘만 만나요

[한 줄 소개]
오늘 준비한 메뉴를 보여드리고, 회원가입한 손님이 메뉴를 담아 주문하면
손님은 "내 주문"에서 자기 주문 상태를 확인하고, 사장님은 관리 화면에서 오늘의 주문을 한눈에 살펴봐요

[핵심 기능 — 7가지]
1. 메뉴 보여주기: 오늘의 메뉴를 카드로 보여드려요 (로그인 없이 누구나 볼 수 있어요)
2. 회원가입·로그인: 손님이 이메일로 가입하고 로그인해요 (Supabase Auth)
3. 주문 담기 & 접수: 로그인한 손님이 메뉴를 담으면 주문이 저장돼요 (Create)
4. 내 주문 보기: 손님은 자기가 남긴 주문과 상태만 확인해요 (Read — 자기 것만!)
5. 사장 전체 조회 & 완료 처리: 사장님만 전체 주문을 보고 "접수 → 완료"로 바꿔요 (Read/Update)
6. 주문 삭제: 사장님이 장난·취소 주문을 정리해요 (Delete)
7. 메뉴 관리: 사장님이 관리 화면에서 메뉴를 등록·수정·삭제하고 품절 여부를 표시해요 (Create/Read/Update/Delete)

[화면 구성 — 경로(라우트) 기준]
■ 주문 페이지 (경로 "/" — src/pages/OrderPage.jsx)
- 상단: 가게 이름 + 장바구니 배지 + (비로그인) [로그인] / (로그인) [내 주문] [로그아웃]
- 중간: 오늘의 메뉴 카드가 쭉 이어져요 — MenuCard 컴포넌트 반복
- 하단: 주문서 — 담은 품목(수량 +/−), 합계, 픽업 희망 시간을 적고 [주문하기]
        (비로그인 상태면 [주문하기] 대신 [로그인하고 주문하기]가 떠요)

■ 회원가입·로그인 (경로 "/auth" — src/pages/AuthPage.jsx)
- shadcn Tabs로 [로그인] / [회원가입]을 오가요 (이메일, 비밀번호)

■ 내 주문 (경로 "/my" — src/pages/MyOrdersPage.jsx)
- 로그인한 손님의 주문만 최신순으로 보여줘요 (품목, 금액, 상태 Badge, 주문 시각)

■ 관리 화면 (경로 "/admin" — src/pages/AdminPage.jsx)  
- 사장님이 로그인하면 → shadcn Tabs로 [주문 관리] / [메뉴 관리] 두 탭을 오가요
- [주문 관리] 탭: 전체 주문 표 + 각 행에 [완료] [삭제] 버튼 + 로그아웃
- [메뉴 관리] 탭: 메뉴 목록 표/카드 + 상품명·가격·설명·품절여부 등록/수정 폼 + 각 행에 [수정] [삭제] 버튼
  (품절 처리는 삭제가 아니라 품절여부 토글로 — 손님 화면에는 품절 표시만 되고 목록에서 사라지지 않아요)
- 사장님이 아닌 로그인 사용자가 "/admin"에 직접 들어오면 "권한이 없습니다" 안내 후 주문 페이지로 이동해요
  (이 화면 가드는 안내용일 뿐이고, 실제 데이터 차단은 아래 RLS 정책이 유일한 방어선이에요)

[데이터 — 저장할 것]
- 주문자(user_id — 로그인한 계정과 자동 연결) / 픽업 희망 시간 (텍스트)
- 주문 내역 (텍스트 — 예: "크루아상 x2, 쑥라떼 x1") / 합계 금액 (숫자)
- 상태 (텍스트, 기본값 "접수") / 주문 시각 (자동)

[orders 테이블 접근 정책 — RLS (반드시 DB 레벨에서 강제)]
- SELECT: 손님은 user_id = auth.uid() 인 행만 조회, 사장은 전체 조회
- INSERT: user_id는 반드시 auth.uid()와 일치해야 함 (WITH CHECK) — 남의 계정 명의로 주문 생성 금지
- UPDATE(상태 변경) / DELETE: 사장만 가능 (손님은 자기 주문도 수정·삭제 불가 — 취소는 사장에게 요청)

[메뉴 데이터 — Supabase menu 테이블로 관리]
- 오늘의 메뉴는 src/data/menu.js 고정 파일이 아니라 Supabase의 menu 테이블에 저장해요
- 컬럼: id / 상품명(name) / 가격(price) / 설명(description) / 품절여부(is_sold_out, 기본값 false) / 등록 시각(created_at)
- 등록/수정/삭제는 관리 화면(/admin)의 [메뉴 관리] 탭에서 사장님이 직접 해요 (Create/Update/Delete)
- 손님·비로그인 방문자는 로그인 여부와 관계없이 메뉴 목록을 조회만 해요 (Read)

[menu 테이블 접근 정책 — RLS (반드시 DB 레벨에서 강제)]
- SELECT: 로그인 여부와 관계없이 누구나 전체 메뉴 조회 가능 (익명 포함)
- INSERT / UPDATE / DELETE: 사장만 가능 (orders 테이블과 동일하게 profiles.role = "owner" 확인)

[회원·권한 설계]
| 역할 | 가입 방법 | 할 수 있는 일 |
|---|---|---|
| 방문자(비로그인) | — | 메뉴만 구경해요 |
| 손님(로그인) | 회원가입 화면에서 직접 | 주문을 남기고, "내" 주문만 확인해요 |
| 사장(로그인) | 계정은 손님과 같은 방식, 권한은 RLS 정책이 부여 | 전체 주문을 보고 완료·삭제하고, 메뉴를 등록·수정·삭제해요 |
※ 손님 A는 손님 B의 주문을 절대 볼 수 없어요 (화면이 아니라 RLS가 보장해요)
※ "사장 가입 버튼"은 없어요 — 사장 지정은 화면 밖(Supabase)에서 해요
※ 역할(role) 저장 위치: profiles 테이블(user_id, role — 기본값 "customer")에 저장해요.
   사장 계정은 가입 후 Supabase에서 관리자가 해당 계정의 role만 "owner"로 직접 수정해요.
※ RLS 정책은 반드시 profiles.role 값을 참조해야 해요. 이메일 하드코딩이나 화면(프론트엔드) 코드의
   role 체크(예: AdminPage.jsx의 if문)로 대체하지 않아요 — 화면의 역할 분기는 안내/UX용일 뿐이고,
   실제 데이터 접근을 막는 유일한 방어선은 RLS예요. 화면 가드만 있고 RLS가 느슨하면, 로그인한 손님이
   개발자도구나 API 직접 호출로 전체 주문 조회·상태 변경·삭제까지 할 수 있게 돼요.

[만들지 않는 것 (범위 제외)]
- 결제 기능 ❌ (주문만 받고, 결제는 픽업할 때 현장에서 해요)
- 비밀번호 찾기/변경, 프로필 수정 ❌ / 소셜 로그인 ❌
- 수량 기반 재고 관리 ❌ (품절여부 on/off만 지원 — 개수 차감·자동 품절 처리는 없어요)
